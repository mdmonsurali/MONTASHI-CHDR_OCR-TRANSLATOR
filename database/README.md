# database/

PostgreSQL data model and SQL files for MONTASHI-CHDR_OCR-TRANSLATOR.

## Data model

One row per uploaded document in the `documents` table (see
[`schema.sql`](./schema.sql)).

| Column         | Type         | Notes                                                  |
|----------------|--------------|--------------------------------------------------------|
| `id`           | UUID         | Primary key, generated by `ocr_service` at upload time |
| `original_name`| TEXT         | The filename the user uploaded                         |
| `status`       | TEXT         | `queued` → `processing` → `ok` or `error`              |
| `scan_type`    | TEXT         | `native` (text-extractable PDF/DOCX) or `scanned`      |
| `page_count`   | INT          | Number of pages processed                              |
| `source_key`   | TEXT         | MinIO object key for the original upload               |
| `markdown_key` | TEXT         | MinIO object key for the extracted markdown            |
| `json_key`     | TEXT         | MinIO object key for the layout JSON                   |
| `docx_key`     | TEXT         | MinIO object key for the reconstructed DOCX            |
| `layout`       | JSONB        | Inline copy of the layout JSON for SQL queries         |
| `elapsed_sec`  | NUMERIC      | Wall-clock time of the OCR pipeline                    |
| `error`        | TEXT         | Failure message (only when `status = 'error'`)         |
| `created_at`   | TIMESTAMPTZ  | When the upload row was inserted                       |
| `updated_at`   | TIMESTAMPTZ  | Last state change                                      |

Bytes live in MinIO under `ocr/documents/{id}/{source,output.md,layout.json,output.docx}`.
The DB only stores keys, not bytes.

## How it gets applied

`ocr_service` copies this folder into its image at build time:

```
COPY database/schema.sql /workspace/database/schema.sql
```

On container start, the FastAPI lifespan handler in
[`ocr_service/src/storage.py`](../ocr_service/src/storage.py) calls
`init_schema()` which reads the file and runs it inside a single transaction.
The DDL is idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`)
so it's safe on every boot.

## Migrations

In v1 there is no migration framework — `schema.sql` is the source of truth and
old columns stay forever. When schema changes become non-trivial:

1. Drop a numbered file in `migrations/` (e.g. `001_add_user_id.sql`).
2. Have `init_schema()` apply migrations in lexical order after `schema.sql`.
3. Track applied migrations in a `schema_migrations` table.

For now `migrations/` is an empty placeholder.

## Inspecting the database

From the host with compose running:

```bash
docker exec -it postgres psql -U ocr -d ocr

# In psql:
\dt                                              -- list tables
\d documents                                     -- describe schema
SELECT id, original_name, status, page_count
  FROM documents ORDER BY created_at DESC LIMIT 20;

SELECT COUNT(*) FILTER (WHERE status='ok')  AS ok,
       COUNT(*) FILTER (WHERE status='error') AS errored
  FROM documents;

-- Query layout JSON: pages with a Table element
SELECT id, original_name
  FROM documents,
       jsonb_array_elements(layout) AS page,
       jsonb_array_elements(page)   AS element
 WHERE element->>'category' = 'Table';
```

## Backup

`pg_data` is a named Docker volume. To back it up:

```bash
docker exec postgres pg_dump -U ocr -d ocr -Fc > ocr.pgdump
# Restore:
cat ocr.pgdump | docker exec -i postgres pg_restore -U ocr -d ocr --clean
```

MinIO bytes are stored separately in the `minio_data` volume; back them up
with `mc mirror` or any S3 backup tool.
